# 2026-07-23 工作日志

## 机器码生成逻辑调整（src/utils/machine.py）

### 改动原因
用户反馈担心换网络导致机器码变化（之前机器码包含 MAC 地址 `uuid.getnode()`）。

### 具体修改
1. **移除 MAC 地址参与机器码计算**：删除 `_get_mac_address()` 函数，`_compute_machine_code()` 现在只用磁盘序列号 + CPU ID + 主机名三项，避免网络/网卡变化影响机器码。
2. **机器码持久化到 proxy_db.db**：
   - 新增 `_load_cached_machine_code()`：从 `ProxyDatabase.get_settings()` 读取 `settings.machine_code` 字段
   - 新增 `_persist_machine_code(code)`：通过 `ProxyDatabase.update_settings({"machine_code": code})` 保存
   - `get_machine_code()` 流程改为：内存缓存 → proxy_db.db 读取 → 重新计算并保存
3. **延迟导入**：`_load_cached_machine_code` 和 `_persist_machine_code` 用 `from ..modules.proxy_server import ProxyDatabase` 延迟导入，避免循环依赖。

### 关键设计点
- 机器码一经写入 proxy_db.db 就稳定不变，即便后续硬件信息读取顺序变化、网卡切换、换网络都不影响
- 已有用户的旧机器码（首次启动时基于含 MAC 的算法生成）会在 proxy_db.db 中被新计算的结果覆盖（因为 settings 里没有 machine_code 字段时会重新计算并保存）
- proxy_db.db 仍是 XXTEA 加密的 JSON，机器码作为 settings 子字段存储

## 403 风控触发 BuddyKey 刷新（src/modules/proxy_server.py）

### 改动
在 403/11140 风控处理分支（约 3248 行）增加 BuddyKey 刷新逻辑，与 429/14018 额度耗尽的处理方式对称：

```python
if resp.status_code == 403 and '"code":11140' in resp_body:
    # ... 原有日志 + mark_key_abnormal ...
    # 新增：尝试获取新 BuddyKey 刷新本地 Key
    new_key = self._refresh_buddykey()
    if new_key:
        tried_key_ids.clear()  # 清除已尝试的 key，让重试循环用新 key
        continue
    # 失败则继续换下一个 Key
```

### 行为
- 上游返回 `{"code":11140,"msg":"request illegal"}` 时，先标记当前 Key 为 abnormal，再向服务端请求新 BuddyKey 替换本地 Key，然后用新 Key 重试请求
- 若获取新 BuddyKey 失败，回退到原有故障转移逻辑（换下一个 Key）
- 复用已有的 `_refresh_buddykey()` 方法（2555 行），该方法会：删除旧的 BuddyKey 上游 key → 添加新的 → 刷新路由器缓存

## BuddyKey 刷新次数限制（src/modules/proxy_server.py）

### 改动
在重试循环外新增计数器 `buddykey_refresh_count`（429/403 共享），上限 `MAX_BUDDYKEY_REFRESH = 3`。

- 429 额度耗尽分支和 403 风控分支在调用 `_refresh_buddykey()` 前都检查计数
- 达到 3 次上限时返回 502 + 提示："多次获取 BuddyKey 仍失败，请检查本地是否开启了代理。开启代理会导致上游地址无法被解析。"
- 计数器在循环外初始化，整个请求生命周期内累加，429 和 403 共享同一个计数

### 迁移逻辑
```python
def _migrate_legacy_db(self, data_dir):
    legacy_path = os.path.join(data_dir, "proxy_db.db")
    if not os.path.exists(legacy_path): return
    if os.path.exists(self._db_path):
        os.remove(legacy_path)  # 新文件已存在，直接删旧
        return
    shutil.copy2(legacy_path, self._db_path)  # 复制
    os.remove(legacy_path)  # 删除旧文件
```

### 行为
- 老用户升级后首次启动：`proxy_db.db` 内容复制到 `proxy_db.key`，旧文件删除
- 新用户：直接使用 `proxy_db.key`
- 迁移在 `__init__` 中 `_load()` 之前执行，确保后续读写都使用新文件
